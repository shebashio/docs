## Overview

ezRMF provides a REST API for programmatic access to all platform features. The API uses JSON request and response bodies, standard HTTP methods, and Bearer token authentication. The base URL for all endpoints is `https://rmf.icbm.dev/api`.

## Authentication

The API supports two authentication methods:

| Method | Use case | Header |
|--------|----------|--------|
| **OIDC session** | Browser-based access | Automatic (cookie-based) |
| **API token** | Programmatic access, CI/CD, automation | `Authorization: Bearer <token>` |

### API tokens

API tokens provide programmatic access with granular permissions. Each token is scoped to a specific set of operations.

| Scope | Description |
|-------|-------------|
| `read` | Read access to projects, controls, evidence, and activities |
| `write` | Create and update projects, controls, and implementation statements |
| `upload` | Upload evidence artifacts |
| `link` | Link evidence to controls |

#### Create a token

```bash
curl -X POST https://rmf.icbm.dev/api/tokens \
  -H "Authorization: Bearer $SESSION_TOKEN" \
  -H "Content-Type: application/json" \
  -d '{
    "name": "ci-pipeline-token",
    "scopes": ["read", "upload", "link"],
    "expires_in_days": 30
  }'
```

<Warning>
The `token` value is only returned once at creation time. Store it securely — you cannot retrieve it again. If lost, revoke the token and create a new one.
</Warning>

## Route tree

The complete API route tree:

```
/api/auth                           # OIDC login/callback/logout
/api/chat                           # POST message, SSE stream
/api/me                             # Current user info
/api/nist                           # NIST 800-53 catalog API
/api/projects                       # CRUD projects
  /:id/conversations                # Chat history
  /:id/data/controls                # NIST controls
  /:id/data/inventory               # HW/SW items
  /:id/data/stakeholders            # Team roles
  /:id/data/poam                    # Findings
  /:id/data/system-info             # System characteristics
  /:id/data/relationships           # Mindmap
  /:id/boe                          # Evidence files
  /:id/emass                        # CCI assessment and export
  /:id/pipeline                     # RMF 7-phase tracking
  /:id/baselines                    # Control baselines
  /:id/skills                       # Agent skills
  /:id/agents                       # Agent configs
  /:id/memories                     # Persistent agent context
  /:id/integrations                 # AWS sync
  /:id/audit-log                    # Change trail
  /:id/usage                        # Token usage tracking
  /:id/mcp/sse                      # MCP tool server (SSE)
```

## Projects

### List projects

```
GET /api/projects
```

Returns all projects accessible to the authenticated user.

### Create a project

```
POST /api/projects
```

**Request body:**

```json
{
  "name": "Mission App ATO",
  "description": "ATO package for the Mission Application",
  "impact_level": "moderate",
  "target_date": "2026-09-30"
}
```

| Field | Type | Required | Description |
|-------|------|----------|-------------|
| `name` | string | Yes | Project name |
| `description` | string | No | Project description |
| `impact_level` | string | Yes | `low`, `moderate`, or `high` |
| `target_date` | string | No | Target authorization date (ISO 8601) |

### Get a project

```
GET /api/projects/{id}
```

Returns the full project object including team members, control summary, and evidence summary.

## Controls

### List project controls

```
GET /api/projects/{id}/data/controls
```

Returns all controls for the specified project with implementation status and CCI counts.

### Update control status

```
PATCH /api/projects/{id}/data/controls/{controlId}
```

**Request body:**

```json
{
  "status": "implemented",
  "implementation_statement": "Account management is handled through Ping Identity..."
}
```

## Evidence (BoE)

### Upload evidence

```
POST /api/projects/{id}/boe
```

Upload a file as evidence. Use `multipart/form-data` encoding.

```bash
curl -X POST https://rmf.icbm.dev/api/projects/{id}/boe \
  -H "Authorization: Bearer $TOKEN" \
  -F "file=@scan-results.ckl" \
  -F "title=RHEL 8 STIG Results" \
  -F "description=Monthly STIG scan for production servers"
```

### Link evidence to controls

```
POST /api/projects/{id}/boe/{eid}/link
```

**Request body:**

```json
{
  "control_ids": ["AC-2", "AC-6", "AU-2"]
}
```

{

## eMASS export

### Export POAM report

```
GET /api/projects/{id}/emass/export
```

Returns an eMASS-formatted Excel file with CUI markings.

### Export CCI test results

```
GET /api/projects/{id}/emass/cci-results
```

Returns per-CCI compliance status for assessment reports.

## Pipeline

### Get pipeline status

```
GET /api/projects/{id}/pipeline
```

Returns the current RMF phase and phase history.

### Advance pipeline phase

```
POST /api/projects/{id}/pipeline/advance
```

Advances the project to the next RMF phase (requires appropriate role).

## Chat and agent

### Send chat message

```
POST /api/chat
```

Sends a message to the AI agent and returns an SSE stream of the response.

### List conversations

```
GET /api/projects/{id}/conversations
```

Returns conversation history for the project.

## Audit log

### List project activities

```
GET /api/projects/{id}/audit-log
```

**Query parameters:**

| Parameter | Type | Default | Description |
|-----------|------|---------|-------------|
| `page` | integer | 1 | Page number |
| `per_page` | integer | 20 | Items per page (max 100) |
| `action` | string | — | Filter by action type |
| `user` | string | — | Filter by user email |
| `from` | string | — | Start date (ISO 8601) |
| `to` | string | — | End date (ISO 8601) |

## Pagination

All list endpoints return paginated results:

| Field | Type | Description |
|-------|------|-------------|
| `items` | array | The result items for the current page |
| `total` | integer | Total number of items across all pages |
| `page` | integer | Current page number (1-indexed) |
| `per_page` | integer | Items per page |

## Error handling

The API returns standard HTTP status codes and a consistent error response format.

| Status code | Description |
|-------------|-------------|
| `400` | Bad request — invalid input or missing required fields |
| `401` | Unauthorized — missing or invalid authentication |
| `403` | Forbidden — insufficient permissions for the requested action |
| `404` | Not found — resource does not exist |
| `409` | Conflict — duplicate resource or invalid state transition |
| `422` | Unprocessable entity — validation error |
| `429` | Too many requests — rate limit exceeded |
| `500` | Internal server error |

**Error response format:**

```json
{
  "error": {
    "code": "VALIDATION_ERROR",
    "message": "impact_level must be one of: low, moderate, high",
    "details": {
      "field": "impact_level",
      "value": "critical"
    }
  }
}
```

## Rate limiting

The API enforces rate limits per authenticated user:

| Tier | Limit | Window |
|------|-------|--------|
| Standard | 100 requests | 1 minute |
| Upload | 20 requests | 1 minute |

When rate limited, the API returns a `429` response with a `Retry-After` header.
