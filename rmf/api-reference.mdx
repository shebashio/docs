## Overview

RMF provides a REST API for programmatic access to all platform features. The API uses JSON request and response bodies, standard HTTP methods, and Bearer token authentication. The base URL for all endpoints is `https://rmf.icbm.dev/api/v1`.

## Authentication

The API supports two authentication methods:

| Method | Use case | Header |
|--------|----------|--------|
| **OIDC session** | Browser-based access | Automatic (cookie-based) |
| **API token** | Programmatic access, CI/CD, automation | `Authorization: Bearer <token>` |

### Scoped API tokens

API tokens provide programmatic access with granular permissions. Each token is scoped to a specific set of operations.

| Scope | Description |
|-------|-------------|
| `read` | Read access to projects, controls, evidence, and activities |
| `write` | Create and update projects, controls, and implementation statements |
| `upload` | Upload evidence artifacts |
| `link` | Link evidence to controls |

#### Create a token

```bash
curl -X POST https://rmf.icbm.dev/api/v1/tokens \
  -H "Authorization: Bearer $SESSION_TOKEN" \
  -H "Content-Type: application/json" \
  -d '{
    "name": "ci-pipeline-token",
    "scopes": ["read", "upload", "link"],
    "expires_in_days": 30
  }'
```

**Response:**

```json
{
  "id": "tok_abc123",
  "name": "ci-pipeline-token",
  "token": "rmf_live_xxxxxxxxxxxxxxxx",
  "scopes": ["read", "upload", "link"],
  "expires_at": "2026-03-30T00:00:00Z",
  "created_at": "2026-02-28T12:00:00Z"
}
```

<Warning>
The `token` value is only returned once at creation time. Store it securely ‚Äî you cannot retrieve it again. If lost, revoke the token and create a new one.
</Warning>

#### List tokens

```bash
curl https://rmf.icbm.dev/api/v1/tokens \
  -H "Authorization: Bearer $SESSION_TOKEN"
```

#### Revoke a token

```bash
curl -X DELETE https://rmf.icbm.dev/api/v1/tokens/{tid} \
  -H "Authorization: Bearer $SESSION_TOKEN"
```

### Token limits

| Setting | Default | Environment variable |
|---------|---------|---------------------|
| Default expiry | 30 days | `DEFAULT_TOKEN_EXPIRY_DAYS` |
| Maximum tokens per user | 10 | `MAX_TOKENS_PER_USER` |

## Projects

### List projects

```
GET /api/v1/projects
```

Returns all projects accessible to the authenticated user.

**Response:**

```json
{
  "items": [
    {
      "id": "proj_abc123",
      "name": "Mission App ATO",
      "impact_level": "moderate",
      "state": "implementation",
      "target_date": "2026-09-30",
      "created_at": "2026-01-15T10:00:00Z"
    }
  ],
  "total": 1,
  "page": 1,
  "per_page": 20
}
```

### Create a project

```
POST /api/v1/projects
```

**Request body:**

```json
{
  "name": "Mission App ATO",
  "description": "ATO package for the Mission Application",
  "impact_level": "moderate",
  "target_date": "2026-09-30"
}
```

**Response:** `201 Created` with the full project object.

| Field | Type | Required | Description |
|-------|------|----------|-------------|
| `name` | string | Yes | Project name |
| `description` | string | No | Project description |
| `impact_level` | string | Yes | `low`, `moderate`, or `high` |
| `target_date` | string | No | Target authorization date (ISO 8601) |

### Get a project

```
GET /api/v1/projects/{id}
```

Returns the full project object including team members, control summary, and evidence summary.

## Evidence

### Upload evidence

```
POST /api/v1/projects/{id}/evidence
```

Upload a file as evidence for a project. Use `multipart/form-data` encoding.

**Request:**

```bash
curl -X POST https://rmf.icbm.dev/api/v1/projects/{id}/evidence \
  -H "Authorization: Bearer $TOKEN" \
  -F "file=@scan-results.ckl" \
  -F "title=RHEL 8 STIG Results" \
  -F "description=Monthly STIG scan for production servers"
```

**Response:** `201 Created`

```json
{
  "id": "ev_xyz789",
  "title": "RHEL 8 STIG Results",
  "description": "Monthly STIG scan for production servers",
  "filename": "scan-results.ckl",
  "size_bytes": 245760,
  "status": "pending",
  "uploaded_by": "user@example.com",
  "created_at": "2026-02-28T14:30:00Z"
}
```

<Info>
Uploading evidence requires the `upload` scope on your API token. Linking evidence to controls requires the `link` scope.
</Info>

### Link evidence to controls

```
POST /api/v1/projects/{id}/evidence/{eid}/link
```

**Request body:**

```json
{
  "control_ids": ["AC-2", "AC-6", "AU-2"]
}
```

**Response:** `200 OK` with the updated evidence object including linked controls.

## Activities

### List project activities

```
GET /api/v1/projects/{id}/activities
```

Returns the activity log for a specific project.

**Query parameters:**

| Parameter | Type | Default | Description |
|-----------|------|---------|-------------|
| `page` | integer | 1 | Page number |
| `per_page` | integer | 20 | Items per page (max 100) |
| `action` | string | ‚Äî | Filter by action type |
| `user` | string | ‚Äî | Filter by user email |
| `from` | string | ‚Äî | Start date (ISO 8601) |
| `to` | string | ‚Äî | End date (ISO 8601) |

**Response:**

```json
{
  "items": [
    {
      "id": "act_001",
      "action": "evidence_uploaded",
      "user": "isso@example.com",
      "details": "Uploaded 'RHEL 8 STIG Results'",
      "timestamp": "2026-02-28T14:30:00Z"
    }
  ],
  "total": 42,
  "page": 1,
  "per_page": 20
}
```

### List user activities

```
GET /api/v1/users/activities
```

Returns all activities performed by the authenticated user across all projects.

## Pagination

All list endpoints return paginated results using the following structure:

| Field | Type | Description |
|-------|------|-------------|
| `items` | array | The result items for the current page |
| `total` | integer | Total number of items across all pages |
| `page` | integer | Current page number (1-indexed) |
| `per_page` | integer | Items per page |

Use `page` and `per_page` query parameters to navigate through results:

```bash
curl "https://rmf.icbm.dev/api/v1/projects?page=2&per_page=50" \
  -H "Authorization: Bearer $TOKEN"
```

## Error handling

The API returns standard HTTP status codes and a consistent error response format.

| Status code | Description |
|-------------|-------------|
| `400` | Bad request ‚Äî invalid input or missing required fields |
| `401` | Unauthorized ‚Äî missing or invalid authentication |
| `403` | Forbidden ‚Äî insufficient permissions for the requested action |
| `404` | Not found ‚Äî resource does not exist |
| `409` | Conflict ‚Äî duplicate resource or invalid state transition |
| `422` | Unprocessable entity ‚Äî validation error |
| `429` | Too many requests ‚Äî rate limit exceeded |
| `500` | Internal server error |

**Error response format:**

```json
{
  "error": {
    "code": "VALIDATION_ERROR",
    "message": "impact_level must be one of: low, moderate, high",
    "details": {
      "field": "impact_level",
      "value": "critical"
    }
  }
}
```

## Rate limiting

The API enforces rate limits per authenticated user:

| Tier | Limit | Window |
|------|-------|--------|
| Standard | 100 requests | 1 minute |
| Upload | 20 requests | 1 minute |

When rate limited, the API returns a `429` response with a `Retry-After` header indicating how many seconds to wait.
