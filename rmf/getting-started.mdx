## Prerequisites

Before deploying RMF, ensure you have the following:

| Requirement | Details |
|------------|---------|
| **Docker** | Docker Engine 24+ and Docker Compose v2 |
| **Node.js** | Node.js 18+ (for CLI tooling) |
| **OIDC provider** | Ping Identity tenant with client credentials |
| **Network access** | Access to the DEDZED platform via [Kasm session](/kasm-workspaces/working-within-kasm) |

<Info>
RMF uses Ping Identity for authentication. If you do not yet have a Ping Identity account, see [Before you begin](/getting-started/before-you-begin) for enrollment instructions.
</Info>

## Deploy with Docker Compose

RMF ships as a Docker Compose stack with environment-based profiles. Clone the repository and configure your environment variables before starting the services.

### 1. Clone the repository

```bash
git clone https://github.com/shebashio/rmf.git
cd rmf
```

### 2. Configure environment variables

Copy the example environment file and update it with your values:

```bash
cp .env.example .env
```

Edit `.env` with the following required variables:

```bash
# Database
DATABASE_URL=postgresql://rmf:rmf_password@db:5432/rmf

# Redis
REDIS_URL=redis://redis:6379/0

# OIDC Authentication (Ping Identity)
OIDC_DISCOVERY_URL=https://auth.pingone.com/<environment-id>/as/.well-known/openid-configuration
OIDC_CLIENT_ID=your-client-id
OIDC_CLIENT_SECRET=your-client-secret

# Activity tracking
ACTIVITY_RETENTION_DAYS=365

# API tokens
DEFAULT_TOKEN_EXPIRY_DAYS=30
MAX_TOKENS_PER_USER=10
```

<Warning>
Never commit your `.env` file to version control. The file contains secrets including your OIDC client credentials and database passwords.
</Warning>

### 3. Start the services

```bash
docker compose up -d
```

This starts four containers:

| Container | Port | Purpose |
|-----------|------|---------|
| `rmf-api` | 8000 | FastAPI application server |
| `rmf-mcp` | 8001 | MCP server for AI integration |
| `rmf-db` | 5432 | PostgreSQL 15 database |
| `rmf-redis` | 6379 | Redis cache and session store |

### 4. Verify the deployment

Check that all services are running:

```bash
docker compose ps
```

Verify the API is responding:

```bash
curl http://localhost:8000/api/v1/health
```

You should receive a `200 OK` response with the service status.

## Configure OIDC with Ping Identity

RMF authenticates users through OIDC with Ping Identity. You need to register RMF as a client application in your Ping Identity environment.

### 1. Create an OIDC application

In your Ping Identity admin console:

1. Navigate to **Applications** and click **Add Application**
2. Select **OIDC Web App** as the application type
3. Set the **Redirect URI** to `https://rmf.icbm.dev/auth/callback`
4. Enable the following scopes: `openid`, `profile`, `email`
5. Save the application and note the **Client ID** and **Client Secret**

### 2. Configure role mapping

RMF maps Ping Identity groups to RMF roles. Create the following groups in Ping Identity and assign users:

| Ping Identity group | RMF role | Description |
|---------------------|----------|-------------|
| `rmf-pm` | PM | Program Manager |
| `rmf-issm` | ISSM | Information System Security Manager |
| `rmf-sca` | SCA | Security Control Assessor |
| `rmf-isso` | ISSO | Information System Security Officer |
| `rmf-ao` | AO | Authorizing Official |

### 3. Update environment variables

Set the `OIDC_DISCOVERY_URL`, `OIDC_CLIENT_ID`, and `OIDC_CLIENT_SECRET` in your `.env` file with the values from step 1, then restart the API:

```bash
docker compose restart rmf-api
```

## Create your first project

Once RMF is deployed and OIDC is configured, you can create your first ATO project.

### 1. Log in to RMF

Navigate to `https://rmf.icbm.dev` in your Kasm browser session. You are redirected to Ping Identity for authentication. After logging in, you land on the RMF dashboard.

### 2. Create a project

1. Click **New Project** on the dashboard
2. Enter a project name (e.g., "Mission App ATO")
3. Select the impact level: **Low**, **Moderate**, or **High**
4. Set the target ATO date
5. Click **Create**

RMF automatically populates the project with the appropriate NIST 800-53 control baseline based on the selected impact level.

### 3. Assign team members

1. Open the project and navigate to the **Team** tab
2. Click **Add Member**
3. Search for users by name or email
4. Assign each team member a role (PM, ISSM, SCA, ISSO, or AO)
5. Click **Save**

<Tip>
Assign at least one user to each role to enable the full evidence approval workflow. See [Roles and permissions](/rmf/roles-and-permissions) for details on what each role can do.
</Tip>

### 4. Review the control baseline

Navigate to the **Controls** tab to see the automatically selected controls. From here you can:

- View control details and implementation guidance
- Mark controls as **Implemented**, **Partially Implemented**, or **Not Implemented**
- Add implementation statements describing how each control is satisfied

### 5. Upload your first evidence artifact

1. Navigate to the **Evidence** tab
2. Click **Upload Evidence**
3. Select a file (e.g., a CKL export from STIGMATE, a policy document, or a screenshot)
4. Add a description and select the control(s) this evidence supports
5. Click **Upload**

The evidence enters the approval workflow and routes to the ISSM for review.
